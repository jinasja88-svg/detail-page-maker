# 멀티 상세페이지 스캔 선택 배치 워크플로우

이 문서는 현재 프로젝트의 멀티 상세페이지 작업 기준 문서다.
이전의 `조합 설계`, `개정안`, `샘플 생성` 문서들은 참고용으로만 두고,
실제 작업 기준은 이 문서 하나로 본다.

핵심은 아래 세 단계다.

1. 전체 스캔
2. 괜찮은 것만 선택
3. 뼈대 길이에 맞춰 배치

현재 기준에서는 여기까지만 확정한다.
즉, 이 문서는 `생성` 이전 단계까지를 다룬다.

---

## 목표

- 상세페이지 URL 여러 개를 수집한다.
- 모든 상세 섹션을 한 번에 훑는다.
- 설득력이 있는 장면만 후보로 뽑는다.
- 전체 흐름은 위에서 아래로 자연스럽게 이어지게 배치한다.
- 최종 길이는 `뼈대 상세페이지`의 섹션 수와 동일하게 맞춘다.
- 같은 소스가 `3개 이상 연속`으로 나오지 않게 한다.

---

## 기본 규칙

### 1. 뼈대의 역할

- 뼈대는 `길이 기준`으로만 사용한다.
- 뼈대가 26개 섹션이면 최종 결과도 26개 슬롯으로 만든다.
- 뼈대 내용을 그대로 많이 쓰는 것이 목적이 아니다.
- 실제 내용은 참조 소스에서 최대한 많이 섞는다.

### 2. 섞는 방식

- 여러 소스에서 좋은 장면만 뽑아 사용한다.
- 단순히 많이 섞는 것이 아니라 `논리적으로 이어지게` 섞는다.
- 같은 소스가 3개 이상 연속으로 나오면 안 된다.
- 연속 2개까지는 허용하되, 다음 슬롯에서는 다른 소스로 끊는다.

### 3. 흐름 규칙

상세페이지는 위에서 아래로 다음 흐름을 우선한다.

- 후킹
- 제품 소개
- 문제 제기
- 해결 제시
- 성능/핵심 기능
- 사용 장면
- 비교/우위 설명
- 편의성
- 마무리

즉, 장면 하나하나가 좋아도 흐름을 깨면 뒤로 밀거나 제외한다.

## 단계별 작업

## Step 1. 전체 스캔

### 입력

- 뼈대 URL 1개
- 참조 URL N개

### 처리

- 각 URL에서 상세 이미지 전부 수집
- 긴 상세 이미지는 섹션 단위로 분할
- 각 소스의 총 섹션 수 기록
- 캡차 실패 소스는 즉시 실패로 표시

### 출력

- `scan_summary.json`
- `section_index.json`

### 현재 판단 기준

- `usable`: 수집과 분할까지 완료된 소스
- `failed`: 캡차 또는 접근 문제로 수집 실패한 소스

### 캡차 처리 원칙

- 우회 로직은 넣지 않는다.
- 캡차가 걸리면 실패로 기록한다.
- 필요하면 나중에 수동 캡처본으로 보강한다.

---

## Step 2. 괜찮은 것만 선택

전체 섹션을 다 쓰지 않는다.
실제로 가치가 있는 장면만 따로 후보로 뽑는다.

### 선택 기준

- 첫 화면 후킹이 강한가
- 문제 제기가 명확한가
- 해결 구조가 잘 보이는가
- 기능 요약이 직관적인가
- 사용 장면이 설득력 있는가
- 비교 장면이 강한가
- 편의성 메시지가 깔끔한가

### 선택 결과 예시

- `HOOK`
- `PAIN`
- `SOLUTION`
- `POWER_FEATURE`
- `USE_SCENE`
- `COMPARE`
- `FEATURE_SUMMARY`
- `TARGET_AREA`
- `BATTERY`
- `LIGHTWEIGHT`
- `EASY_USE`
- `CHARGING`

### 출력

- `selection_pool.json`

이 파일에는 아래를 담는다.

- `section_id`
- `source`
- `file`
- `role`
- `reason`

---

## Step 3. 뼈대 길이에 맞춰 배치

### 핵심

- 최종 슬롯 수는 뼈대 길이와 동일
- 선택된 장면을 논리적으로 배치
- 부족한 슬롯만 최소한으로 뼈대 유지

### 배치 규칙

- 전체 길이 = 뼈대 길이
- 좋은 장면은 앞쪽 핵심 슬롯에 우선 배치
- 같은 소스 3연속 금지
- 중후반부에서도 다른 소스로 흐름을 환기
- keep 슬롯은 비는 자리를 메우는 용도로만 사용

### 출력

- `ordered_plan_v2.json`
- `review.html`
- `ordered_plan_gallery.html`
- `ordered_plan_stacked.html`

---

## 현재 산출물 설명

### 1. `review.html`

- 스캔 결과 요약
- 선택된 후보 요약
- 26개 슬롯 정렬 요약

### 2. `ordered_plan_gallery.html`

- 26개 슬롯을 이미지 썸네일 목록으로 확인
- 각 슬롯의 `source`, `role`, `type`, `section_id` 확인 가능

### 3. `ordered_plan_stacked.html`

- 26개 슬롯을 상세페이지처럼 위에서 아래로 쭉 이어서 확인
- 실제 상세 섹션을 순서대로 내려보며 흐름 검토 가능

---

## 현재 적용된 26개 배치 원칙

- 뼈대 길이 `26`
- 핵심 선택 소스 `A`, `C`, `D`, `E`
- 실패 소스 `B`, `F`
- 초반은 `후킹 -> 소개 -> 문제 -> 해결`
- 중반은 `기능 -> 활용 -> 비교 -> 편의`
- 후반은 `보강 장면 + 최소 keep`

즉, 현재 기준은
`뼈대는 길이 기준`
`내용은 가능한 한 여러 소스에서 선정`
`정렬은 설득 흐름 기준`
이다.
