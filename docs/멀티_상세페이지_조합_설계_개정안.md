# 멀티 상세페이지 조합 시스템 설계 개정안

> 여러 경쟁사 상세페이지를 참고하되, 원문 문구와 원본 레이아웃을 직접 복제하지 않고
> "설득 구조", "정보 구조", "톤앤매너"만 추출해 내 상품 기준의 신규 상세페이지를 만드는 방식.

---

## 개정 목표

- 기존 단일 상페 재현형 방식에서 벗어나 멀티 소스 조합형으로 확장
- 1번 상페를 전체 톤앤매너의 기준으로 삼되, 다른 상페의 장점은 역할 단위로 흡수
- 문구는 변형 수준이 아니라 **재작성** 기준으로 운영
- 디자인은 복붙이 아니라 **재해석된 통일 디자인**으로 출력
- 저작권 리스크를 줄이기 위해 원문/원레이아웃 직접 재사용을 금지

---

## 핵심 원칙

### 1. 참고는 하되 복제는 하지 않는다
- 경쟁사 상세페이지는 "아이디어 소스"로만 사용
- 최종 결과물은 내 상품 기준의 신규 카피와 신규 HTML로 생성
- 원문 문구, 표/도형 구성, 배치 구조를 그대로 재현하지 않음

### 2. 뼈대와 소스를 분리한다
- `뼈대(A)`는 디자인 시스템 기준
- `참조(B~N)`는 설득 포인트, 표현 아이디어, 정보 구성 소스
- 최종 결과는 `A의 톤앤매너 + B/C의 강점 메시지 + 내 상품 정보`의 조합

### 3. 섹션 이미지를 고르는 것이 아니라 섹션 역할을 재구성한다
- 단순히 "어느 장면을 쓸까"가 아니라
- "이 장면이 무슨 역할을 했는가"를 먼저 추출
- 이후 역할별 최적 구조를 새 순서로 재배열

### 4. 카피는 변형이 아니라 재작성이다
- 원본 문장을 조금 손보는 수준은 불충분
- 핵심 메시지와 설득 구조만 유지하고 문장은 새로 작성
- 3어절 이상 연속 동일 표현 금지

### 5. 디자인은 통일이 아니라 재해석까지 포함한다
- 뼈대 상페의 배색, 간격, 시선 흐름은 유지
- 다른 상페 요소를 섞을 때도 동일한 디자인 시스템 안에서 다시 제작
- 결과물은 "A를 닮았지만 다른 페이지"가 되어야 함

## 기존 설계 대비 핵심 변화

| 항목 | 기존안 | 개정안 |
|------|------|------|
| 조합 기준 | 역할별 베스트 섹션 선택 | 역할별 베스트 "설득 요소" 선택 |
| 카피 처리 | 변형 | 재작성 |
| 디자인 처리 | 1번 상페 톤앤매너 통일 | 1번 상페 기반 재해석 디자인 시스템 |
| 저작권 대응 | 변형 필수 | 직접 복제 금지 + 유사도 점검 추가 |
| 선택 단위 | 섹션 이미지 중심 | 역할, 메시지, 구조, 시각 패턴 분리 |
| 검수 포인트 | 선택/배열/카피 검토 | 선택/배열/카피/유사도/증빙 검토 |

## 전체 파이프라인

```text
입력: URL 3~5개 + 내 상품 정보 + 제품 사진
  ↓
Phase 1. 멀티 크롤링 + 분할
  ↓
Phase 2. 전체 섹션 역할 태깅
  ↓
Phase 3. 설득 요소 추출
  ↓
Phase 4. 역할별 베스트 구조 선택
  ↓
Phase 5. CRO 기반 최적 배열
  ↓
Phase 6. 카피 재작성
  ↓
Phase 7. 디자인 시스템 추출 + 재해석
  ↓
Phase 8. 제품 사진 적용 + HTML 출력
  ↓
Phase 9. 저작권/유사도/흐름 검수
```

## Phase 1. 멀티 크롤링 + 분할

### 목적
- 뼈대용 상세페이지 1개와 참조용 상세페이지 2~4개를 수집
- 모든 상세페이지를 동일 기준으로 섹션 단위 분할

### 입력 예시
```yaml
urls:
  - url: "https://smartstore.naver.com/aaa/products/111"
    label: "A"
    role: "뼈대"
  - url: "https://smartstore.naver.com/bbb/products/222"
    label: "B"
    role: "참조"
  - url: "https://www.coupang.com/vp/products/333"
    label: "C"
    role: "참조"
```

### 처리
- `crawler.py`를 URL 리스트에 대해 순차 실행
- `image_splitter.py`로 모든 상세이미지를 섹션 단위로 정리
- 각 소스별 디렉터리와 메타데이터 기록

### 출력
- `config.json`
- `sources/*/detail_images/*.jpg`
- `sources/*/sections/*.png`

## Phase 2. 전체 섹션 역할 태깅

### 목적
- 각 섹션이 상세페이지에서 수행하는 역할을 분류
- 이후 선택, 재배열, 재작성의 기준을 마련

### 역할 태그
- `HOOK`
- `PAIN`
- `SOLUTION`
- `FEATURE`
- `BENEFIT`
- `SOCIAL_PROOF`
- `HOW_TO`
- `COMPARE`
- `INGREDIENT`
- `BRAND`
- `BUNDLE`
- `GUARANTEE`
- `CTA`
- `FILLER`

### 필수 추출 필드
- `role`
- `copy_text`
- `copy_score`
- `design_score`
- `core_message`
- `visual_pattern`
- `proof_type`

### 출력
- `section_analysis.json`

## Phase 3. 설득 요소 추출

### 목적
- 섹션 자체를 고르는 것이 아니라, 그 섹션의 장점을 분해해서 확보

### 각 섹션에서 추출할 요소
- `핵심 메시지`
- `설득 구조`
- `정보 밀도`
- `증빙 방식`
- `시각 패턴`
- `감정 톤`

### 설득 구조 예시
- 문제 제기 → 해결 제시
- 숫자 제시 → 효과 강조 → 증명
- 후기 제시 → 신뢰 강화 → 구매 유도
- 성분 설명 → 효능 설명 → 사용 장면 연결

### 출력 예시
```json
{
  "id": "B_003",
  "role": "HOOK",
  "core_message": "짧은 기간 안의 변화 강조",
  "persuasion_structure": "숫자 제시 -> 효과 강조 -> 감각적 증명",
  "visual_pattern": "좌측 카피 우측 제품",
  "proof_type": "기간 강조형 주장"
}
```

### 출력
- `persuasion_map.json`

---

## Phase 4. 역할별 베스트 구조 선택

### 목적
- 역할마다 가장 강한 설득 논리와 가장 적합한 시각 패턴을 선택

### 선택 단위
- `copy_source`: 카피 아이디어 소스
- `proof_source`: 근거/증빙 아이디어 소스
- `visual_source`: 시각 구조 소스
- `skeleton_match`: A와 얼마나 잘 맞는지

### 선택 규칙
- 뼈대 A와 충돌이 적은 구조 우선
- `FILLER` 자동 제외
- 같은 역할의 후보가 많으면 1순위/2순위까지 저장
- 후기/비교/수치성 주장은 실제 검증 가능한 내용만 채택

### 출력
- `best_structure_selection.json`

## Phase 5. CRO 기반 최적 배열

### 목적
- 단순 원본 순서가 아니라 구매 전환 흐름 기준으로 재배열

### 기본 템플릿
```text
1. HOOK
2. PAIN
3. SOLUTION
4. FEATURE
5. BENEFIT
6. SOCIAL_PROOF
7. INGREDIENT
8. HOW_TO
9. COMPARE
10. BRAND
11. BUNDLE
12. GUARANTEE
13. CTA
```

### 배열 원칙
- A의 전체 흐름을 기본 참고
- 단, CRO 우선순위가 더 타당하면 CRO 우선
- 역할 누락은 허용하되 `HOOK`, `SOLUTION`, `FEATURE`, `CTA`는 가능하면 유지
- 유사한 역할이 중복되면 정보 밀도와 설득 강도 기준으로 정리

### 출력
- `arrangement.json`
- `arrangement_review.html`

## Phase 6. 카피 재작성

### 목적
- 원문 문장을 버리고 내 상품 기준의 신규 카피 생성

### 재작성 규칙
- 원문 문장 재사용 금지
- 3어절 이상 연속 동일 금지
- 경쟁사 제품명/수치 제거
- 내 상품의 실제 특징, 성분, 효능, 타겟에 맞게 다시 작성
- 의료적/과장 표현은 별도 검수

### 재작성 입력값
- `role`
- `core_message`
- `persuasion_structure`
- `product_info`
- `target_customer`
- `tone_guideline`

### 출력
- `copy_rewrite.json`

### 출력 예시
```json
{
  "role": "HOOK",
  "source_ids": ["A_013", "B_003"],
  "core_message": "짧은 사용 기간 안의 체감 변화",
  "new_copy": "매일 바르는 보습, 며칠 지나면 피부결이 먼저 달라집니다",
  "structure": "기간 암시 -> 변화 강조 -> 체감 포인트"
}
```

## Phase 7. 디자인 시스템 추출 + 재해석

### 목적
- A 상페의 분위기를 기준으로 전체 디자인 언어를 통일
- 동시에 직접 복제처럼 보이지 않도록 새로운 레이아웃으로 재해석

### A에서 추출할 항목
- 메인/보조 배경색
- 강조색
- 제목/본문 타이포 규칙
- 섹션 간 여백
- 주요 레이아웃 패턴
- 아이콘/배지/박스 스타일
- CTA 스타일

### 재해석 원칙
- 동일한 배색 계열 사용 가능
- 동일한 정보 흐름은 허용
- 동일한 표/박스/문장 배치 재현은 지양
- 시각 패턴은 유지하되 세부 구성은 새로 만듦

### 출력
- `design_system.json`
- `section_render_plan.json`

## Phase 8. 제품 사진 적용 + HTML 출력

### 목적
- 재작성된 카피와 재해석된 디자인 시스템을 결합해 최종 HTML 생성

### 처리
- 내 제품 사진을 섹션별 용도에 맞게 배치
- 기존 HTML 재현 구조를 재사용하되 멀티 소스용 파라미터 추가
- 텍스트는 `contenteditable` 유지

### 출력물
- `section_*_compare.html`
- `combined_review.html`
- `final_page.html`
- `text_layers.json`

## Phase 9. 저작권/유사도/흐름 검수

### 목적
- 결과물이 경쟁사 페이지와 지나치게 유사하지 않은지 확인
- 카피와 디자인 모두 최종 리스크 점검

### 검수 항목

#### 카피 검수
- 원문과 3어절 이상 연속 일치 여부
- 핵심 헤드카피 유사 표현 존재 여부
- 경쟁사 고유 문장 구조 잔존 여부

#### 디자인 검수
- 특정 섹션의 박스 구조, 표 구조, 타이포 배치가 과도하게 유사한지 점검
- A 상페와 톤앤매너만 유사하고 결과물은 독립적인지 확인

#### 주장 검수
- 후기형 문구, 수치형 문구, 비교형 문구가 실제 근거와 맞는지 확인
- 검증 불가 표현은 삭제 또는 일반화

### 출력
- `similarity_report.json`
- `compliance_checklist.md`

## 운영 원칙 요약

- 경쟁사 상세페이지는 참고자료이지 원본이 아니다
- 최종 산출물은 새 카피, 새 구조, 새 HTML이어야 한다
- 1번 상페는 뼈대이지만 복제 대상이 아니다
- 여러 상페를 섞되 결과물은 한 브랜드의 페이지처럼 보여야 한다
- 설득력보다 먼저 사실성, 통일감, 리스크 안전성을 확보한다
