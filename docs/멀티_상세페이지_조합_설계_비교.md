## 단계별 비교

### 1. 수집 단계

기존안
- 멀티 URL 수집과 분할에 초점

개정안
- 동일
- 차이는 이후 단계를 위해 소스별 메타데이터와 역할 구분을 더 엄격히 남김

판단
- 수집 단계는 큰 차이 없음
- 기존안 구조를 그대로 재사용 가능

### 2. 분석 단계

기존안
- 섹션 역할 태깅
- 카피 점수, 디자인 점수 부여

개정안
- 역할 태깅은 유지
- 여기에 설득 구조, 증빙 방식, 감정 톤, 시각 패턴까지 추가 추출

판단
- 개정안이 후속 단계 품질을 더 잘 받쳐줌
- 단순 점수만으로는 안전한 재조합이 어려움

### 3. 선택 단계

기존안
- 카피 베스트와 디자인 베스트를 섹션 단위로 선택

개정안
- 카피 아이디어, 증빙 아이디어, 시각 구조를 분리해서 선택

판단
- 기존안은 강력하지만 섹션 복제 느낌이 남을 수 있음
- 개정안은 더 번거롭지만 결과물이 더 독립적임

### 4. 배열 단계

기존안
- CRO 기반 순서로 배열

개정안
- CRO 기반은 유지
- 단, 중복 정보 제거와 필수 역할 유지 원칙을 더 명확히 둠

판단
- 배열 자체는 기존안도 충분히 좋음
- 개정안은 운영 규칙이 더 명확함

### 5. 카피 단계

기존안
- 원문을 변형해서 사용

개정안
- 원문을 버리고 핵심 메시지/구조만 가져와 새 문장 작성

판단
- 이 차이가 가장 큼
- 저작권 리스크를 줄이려면 개정안 방식이 더 적합

### 6. 디자인 단계

기존안
- A 상페 스타일로 통일

개정안
- A 상페 스타일을 디자인 시스템으로 추출
- 그대로 따지 않고 재해석 규칙을 적용

판단
- 기존안은 빠름
- 개정안은 더 안전하고 브랜드 완성도가 높음

### 7. 검수 단계

기존안
- 사용자 확인 중심

개정안
- 사용자 확인 + 유사도/주장 검수 리포트 추가

판단
- 실제 운영에서는 개정안이 더 실무적
- 특히 문구, 수치, 후기성 표현 관리에 유리

## 어떤 경우에 무엇을 쓰면 좋은가

### 기존안이 맞는 경우
- 빠르게 프로토타입을 만들고 싶을 때
- 일단 멀티 소스 조합 파이프라인부터 돌려보고 싶을 때
- 내부 테스트용으로 구조만 검증할 때

### 개정안이 맞는 경우
- 실제 판매용 상세페이지를 만들 때
- 저작권 리스크를 줄이고 싶을 때
- 브랜드 일관성과 완성도를 더 높이고 싶을 때
- 장기적으로 자동화 시스템으로 굴릴 때

## 추천 결론

실제 새 버전의 기준 문서는 개정안으로 가는 편이 맞습니다.

이유:
- 기존안의 장점은 유지한다
- 사용자 아이디어였던 "여러 상페의 장점 조합"을 더 정교하게 반영한다
- 단순 조합이 아니라 신규 생성형 방식으로 전환되어 저작권 리스크를 더 낮춘다
- 앞으로 코드 자동화할 때도 `역할`, `메시지`, `증빙`, `시각 패턴` 단위로 분리해 다루는 쪽이 확장성이 높다

- 문서 기준은 개정안 채택
- 기존안은 1차 구조 참고용으로 유지
- 구현은 기존안의 Phase 구조를 최대한 살리되

- 겉 구조는 기존안
